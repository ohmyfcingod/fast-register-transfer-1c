# Быстрый перенос регистров (БПР)

![License: MIT](https://img.shields.io/badge/License-MIT-green.svg)
![Platform: 1C:Enterprise 8.5](https://img.shields.io/badge/1C%3AEnterprise-8.5-blue.svg)
![Delivery: .cfe extension](https://img.shields.io/badge/delivery-.cfe%20AddOn-orange.svg)

> **TL;DR (EN).** Fast binary transfer of register record sets (information / accumulation / accounting registers) between 1C:Enterprise infobases. Record sets are serialized into binary chunks plus a `manifest.json`, moved as files, and loaded back with recorder-level replacement — orders of magnitude faster than XML/XDTO exchange plans. Ships as a self-contained configuration extension (`БПР_` prefix, AddOn, compatibility mode 8.5.1) with zero dependencies on the host configuration. Measured throughput: ~90–180k rows/s export, ~70–130k rows/s import; a 5-million-row round-trip passes a 1:1 data check. Parallel background-job workers, optional deflate compression, portable JSON chunk format, live progress with cancellation, and resume after interruption are built in.

## Что это и зачем

Штатные механизмы обмена 1С (планы обмена, выгрузка через XML/XDTO-сериализацию) устроены универсально и потому медленно: построчная сериализация в текст, оверхед на разбор, одна нитка. Когда задача — перелить миллионы строк регистров между базами с идентичной конфигурацией (клон-синк, наполнение тестового контура, перенос архива движений), эта универсальность не нужна.

БПР решает ровно эту задачу: наборы записей регистров (сведений, накопления, бухгалтерии) сериализуются в **бинарные чанки** (`ЗначениеВФайл`) с описанием в `manifest.json`, переносятся файлами (или через шину — см. roadmap) и загружаются **замещением по регистратору** (независимые РС — дозаписью). Движок работает с любыми регистрами по именам-параметрам.

Поставка — **расширение конфигурации**: префикс `БПР_`, назначение `AddOn`, режим совместимости 8.5.1, **ноль зависимостей** от базовой конфигурации. Подключается к любой конфигурации (включая БСП-релизы) без правки основной. Всего 7 объектов метаданных.

## Ключевые возможности

- **Бинарные чанки + `manifest.json`** — набор записей режется на порции ~100–200 тыс. строк (размер подобран замером), каждая пишется одним `ЗначениеВФайл`; манифест описывает порции, версию, сжатие и отбор.
- **Параллельная выгрузка/загрузка фоновыми заданиями** — диапазон режется по MIN/MAX с дешёвым `КОЛИЧЕСТВО` на поток; выгрузка масштабируется до ×2,2 на 8 потоках. Запуск заданий адаптивный: есть БСП — через `ОбщегоНазначения.ВыполнитьМетодКонфигурации`, нет — через собственный тонкий шим; отдельное ветвление для файловой и клиент-серверной ИБ.
- **Опциональное сжатие deflate** — чанк через `ХранилищеЗначения` с deflate=9: ~15% от исходного размера (151,5 МБ → 22,4 МБ на 1 млн строк) ценой ×3,8 к записи. Для локального клон-синка выгоднее без сжатия, для передачи по сети — со сжатием.
- **Портативный JSON-формат чанков** — параметр `ФорматДанных` (0 = bin, 1 = JSON через `СериализаторXDTO` с явным назначением типов). JSON переживает смену версии платформы (страховка от версионной хрупкости `.bin`); формат фиксируется в манифесте, загрузка распознаёт его по расширению файла.
- **Живой прогресс с отменой** — перенос идёт фоновым заданием, воркеры пишут `_прогресс_*.json` после каждого чанка, форма опрашивает раз в 2 с и рисует HTML-прогресс; кнопка «Отмена»; план в `_план.json`, итог в `_итог.json`.
- **Отбор по периоду** — `НачалоПериода`/`КонецПериода` для РН/РБ и периодических РС; отбор фиксируется в манифесте; движок запрещает загрузку частичной выгрузки с полной очисткой (только дозапись / замещение по регистратору).
- **Профили производительности СУБД** — пресеты под файловую и клиент-серверную ИБ (для MS SQL чанк загрузки держать < 5000 строк — иначе эскалация блокировок до table lock).
- **Докачка после обрыва** — sidecar-файл `_загружено_<Регистр>.txt`; повтор с `ОчищатьПередЗагрузкой = Ложь` пропускает уже загруженные чанки. Запись чанка атомарна, замещение по регистратору идемпотентно.
- **Метрики OpenMetrics** — `bpr_transfer.prom` (`bpr_rows_total`, `bpr_seconds`, `bpr_rows_per_second`, `bpr_bytes_total`, `bpr_threads`) рядом с отчётами; подключается к Grafana через textfile collector `node_exporter`.
- **Самотест** — встроенный нагрузочный харнесс: синтетическое наполнение, бенчмарки, `ПолныйТест` с HTML-отчётом. На реальных регистрах тест выполняет только выгрузку-чтение, данные не меняются.

## Производительность

Два стенда: **файловая ИБ** (NVMe, автономный сервер) и **клиент-серверная ИБ (MS SQL Server, SSD)**. Регистр сведений, синтетические данные ~160 байт/строку.

### Одиночный поток

| Объём | Выгрузка | Загрузка | Размер | Сверка |
|---|---|---|---|---|
| 1 млн строк | 9,4 с (106k стр/с) | 7,8 с (128k стр/с) | 151,5 МБ | 1:1 OK |
| 5 млн строк | 56,8 с (88k стр/с) | 53,1 с (94k стр/с) | 772,9 МБ | 1:1 OK |

### Параллельная выгрузка (5 млн строк)

| Потоков | Время | Скорость | Ускорение |
|---|---|---|---|
| 1 | 52,8 с | 94,7k стр/с | ×1,00 |
| 2 | 45,8 с | 109k стр/с | ×1,15 |
| 4 | 27,8 с | 180k стр/с | ×1,90 |
| 8 | 24,0 с | 208k стр/с | ×2,20 |

### Параллельная загрузка (5 млн строк)

| Потоков | Время | Скорость | Ускорение |
|---|---|---|---|
| 1 | 89,1 с | 56,1k стр/с | ×1,00 |
| 4 | 71,9 с | 69,6k стр/с | ×1,24 |
| 8 | 71,3 с | 70,1k стр/с | ×1,25 |

Выгрузка параллелится хорошо; на файловой СУБД плато после ~4 потоков — упор в read-lock СУБД, не в диск. Загрузка на файловой ИБ упирается в монопольную запись `.1CD` (~×1,25). Полный round-trip 5 млн строк проходит сверку 1:1.

### Клиент-серверная СУБД: MS SQL Server на SSD (5 млн строк)

На серверной СУБД снимаются оба файловых потолка — и чтение, и, главное, **запись**:

| Потоков | Выгрузка | Ускорение | Загрузка | Ускорение |
|---|---|---|---|---|
| 1 | 98,1 с (51k стр/с) | ×1,00 | 291,0 с (17,2k стр/с) | ×1,00 |
| 2 | 54,8 с (91k стр/с) | ×1,79 | 163,2 с (30,6k стр/с) | ×1,78 |
| 4 | 32,8 с (153k стр/с) | ×2,99 | 116,1 с (43,1k стр/с) | ×2,51 |
| 8 | **20,8 с (241k стр/с)** | **×4,72** | **68,7 с (72,8k стр/с)** | **×4,24** |

Выгрузка масштабируется почти линейно до 8 потоков; **запись тоже параллелится** (×4,24 против ×1,25 на файловой) — дозапись независимых РС непересекающимися ключами в режиме обмена не конфликтует и на MS SQL.

Два практических вывода с этого стенда:
- **Размер чанка для загрузки решает**: полный round-trip с чанком 50 000 грузится в ~1,7 раза быстрее, чем с чанком 4 000 (115,9 с против 200,4 с на 4 потоках). Пресет профиля «MS SQL Server» держит чанк ниже порога эскалации блокировок (~5 000 строк) — это цена за то, чтобы заливка не лочила таблицу целиком для других пользователей. В ночное окно, когда база ваша, ставьте чанк крупнее вручную.
- **Для выгрузки на SQL крупный чанк чуть лучше**: подбор чанка дал минимум на 500 000 (91,4 с против ~100 с на 25 000–100 000); плоская зона широкая, дефолт 100 000 остаётся разумным.

Матрица форматов на этом же стенде (1 поток): bin 98/473 с; deflate9 126/508 с (112 МБ); json 133/561 с (1,73 ГБ) — соотношение форматов то же, что на файловой; абсолютное время загрузки — следствие мелкого профильного чанка, см. выше.

### Сжатие (1 млн строк)

| Вариант | Запись | Размер | Чтение |
|---|---|---|---|
| bin без сжатия | 1,21 с | 151,5 МБ | 2,09 с |
| bin + deflate=9 | 4,60 с | 22,4 МБ (14,8%) | 2,27 с |

### Матрица форматов транспорта (5 млн строк, 1 поток, файловая ИБ, полный round-trip)

| Вариант | Выгрузка | Размер | Загрузка | Сверка |
|---|---|---|---|---|
| bin | 84,0 с (59,5k стр/с) | 772,9 МБ (162 байт/строку) | 61,2 с (81,7k стр/с) | OK |
| bin + deflate=9 | 108,0 с (46,3k стр/с) | **112,2 МБ (14,5%)** | 81,0 с (61,8k стр/с) | OK |
| json | 118,8 с (42,1k стр/с) | 1,73 ГБ (×2,3 к bin) | 188,6 с (26,5k стр/с) | OK |

Вывод: **bin** — рабочая лошадка (быстрее всех), **deflate** — когда важен размер/сеть (в 7 раз меньше ценой ~×1,3 по времени), **json** — страховка переносимости между версиями платформы (×2,3 размер, ×3 время загрузки — цена portable-режима). Кнопка «Матрица форматов» в режиме разработчика повторяет замер на вашем стенде (`benchmark_format.txt` + `benchmark_format.prom`).

Дефолты (размер чанка, сжатие, число потоков) выбраны по замерам, не предположениям.

## Установка

Артефакт — `build/БыстрыйПереносРегистров.cfe`.

1. **Подключить расширение.** Конфигуратор → Конфигурация → Расширения конфигурации → Добавить → выбрать `.cfe`.
2. **Доверенный режим — ОБЯЗАТЕЛЕН.** Снять галки **«Безопасный режим»** и **«Защита от опасных действий»**, перезапустить. Иначе файловый I/O (`ЗначениеВФайл`) падает с «Установлен безопасный режим. Выполнение операции запрещено» — запись в регистры работает, а выгрузка в файл нет. Кодом изнутри это не обходится, это требование поставки.
   - Без Конфигуратора: веб-клиент → «Режим технического специалиста» → «Функции для технического специалиста» → «Управление расширениями конфигурации» → выключить оба тумблера → «Перезапустить».
3. **Интерфейс целевой базы — 8.5** (`interfaceCompatibilityMode = Version8_5`), иначе форма откроется пустой (заголовок есть, тело без элементов). В БСП-релизах обычно уже так. Лечение в `Configuration.mdo`: перед `<compatibilityMode>` добавить `<interfaceCompatibilityMode>Version8_5</interfaceCompatibilityMode>` и `<version85InterfaceMigrationMode>Use</version85InterfaceMigrationMode>`.
4. **Файловая ИБ и EDT.** Применение расширения к файловой ИБ напрямую из EDT (`update_database`) может падать с `Файл не обнаружен 'zip:///'` (дефект среды, не расширения). Рабочие обходы: `.cfe` через Конфигуратор, либо **автономный сервер** — он обслуживает файловую ИБ по пути и применяет расширения штатно.
5. **Проверка.** Обработка «Быстрый перенос регистров» (подсистема «Сервис») → вкладка «Бенчмарк и настройки» → `ПолныйТест` → HTML-отчёт `full_test_report.html` и файлы `*.bin` / `manifest.json` в папке обмена.

## Быстрый старт

1. В **базе-источнике** открыть обработку «Быстрый перенос регистров» (подсистема «Сервис»).
2. Выбрать регистры (по именам), указать папку обмена, при необходимости — период, формат (bin/JSON), сжатие, число потоков. Нажать **«Выгрузить»**.
3. Перенести папку обмена на приёмник (или указать общую).
4. В **базе-приёмнике** открыть ту же обработку, указать папку, нажать **«Загрузить»**. Прогресс отображается живьём, есть отмена.
5. При обрыве — повторить загрузку с `ОчищатьПередЗагрузкой = Ложь`: уже загруженные чанки будут пропущены по sidecar-файлу.

## Формат транспорта

Папка обмена после выгрузки:

```
<папка обмена>/
├── manifest.json                  # версия, регистры, порции, формат (bin/json), сжатие, отбор
├── <Регистр>_0001.bin             # чанк: ТаблицаЗначений через ЗначениеВФайл (формат 0)
├── <Регистр>_0002.bin
├── <Регистр>_0001.json            # чанк в JSON (СериализаторXDTO, формат 1)
├── _загружено_<Регистр>.txt       # sidecar докачки: какие чанки уже загружены приёмником
├── _план.json                     # план переноса (пишется при старте)
├── _прогресс_<N>.json             # живой прогресс воркеров (после каждого чанка)
├── _итог.json                     # итог переноса
└── bpr_transfer.prom              # метрики OpenMetrics для Grafana/node_exporter
```

Детали формата:

- Чанк — набор записей одного регистра; подчинённые (документам) регистры выгружаются с `УПОРЯДОЧИТЬ ПО Регистратор`, граница чанка режется **только на смене регистратора** — движения одного регистратора не разрываются между чанками.
- Загрузка: независимые РС — дозаписью, подчинённые регистры — замещением наборов по регистратору (идемпотентно).
- Манифест хранит поля `Версия`, `Сжатие`, `Формат`, `Отбор` — загрузка самодостаточна по манифесту.

## Ограничения

- **Идентичные базы.** Ссылки (GUID) между разными базами не маппятся: инструмент рассчитан на базы с идентичной конфигурацией и общими ссылочными данными (клон, копия, синхронизированные справочники). JSON-формат решает cross-**версию** платформы, но не cross-**базу** — маппинг ссылок это отдельная задача, которую БПР не решает.
- **`.bin` — внутренний формат платформы** (`ЗначениеВФайл`): может быть несовместим между релизами платформы и при изменении метаданных. Для версионной устойчивости используйте JSON-формат чанков.
- Загрузка по умолчанию однопоточная; параллелится в первую очередь выгрузка. На файловой СУБД параллелизм выгрузки выходит на плато после ~4 потоков.
- MS SQL: чанк загрузки держать < 5000 строк, иначе эскалация блокировок до table lock.
- Отбор по периоду выгружается последовательно (без параллельного сплита — сознательно).
- Расширение исполняется доверенным кодом от имени серверного процесса (rphost); встроена валидация пути папки обмена (запрет пустого/относительного пути, корня диска, системных каталогов).

## Roadmap

### Гистограммная балансировка потоков (идея, сознательно не реализована)

Текущий сплит диапазона — MIN/MAX с дешёвым `КОЛИЧЕСТВО` на поток (даёт ~×1,6–1,9); реальное число строк на поток пишется в лог, так что перекос данных виден. «Теоретически правильные» квантили через полную сортировку были проверены и отвергнуты замером: полный `УПОРЯДОЧИТЬ ПО` на 5 млн строк убил параллелизм (×1,06). Следующий шаг — гистограмма без сортировки:

```bsl
ВЫБРАТЬ ГОД(Период) КАК Год, КОЛИЧЕСТВО(*) КАК Строк
ИЗ РегистрНакопления.<Имя>
СГРУППИРОВАТЬ ПО ГОД(Период)
```

— веса лет → резать потоки по равному **весу** строк, а не по ширине диапазона дат. Даёт выигрыш в основном на серверной СУБД с перекошенными данными; включать по факту перекоса, который виден в логе «строк/поток».

### RabbitMQ / шина (шов готов, коннектор за пользователем)

Транспорт чанков абстрагирован; все границы помечены в модуле менеджера тегом `ТОЧКА РАСШИРЕНИЯ` (находятся грепом). Движок чанкования/замещения/параллелизма при замене транспорта не трогается.

| Граница | Функция движка | RabbitMQ-аналог |
|---|---|---|
| выход, чанк | `ЗаписатьПорцию` / `ЗаписатьПорциюПотока` | `publish` (routing key = имя регистра) |
| выход, манифест | `ЗаписатьМанифест` | стартовое сообщение / свойства |
| вход, манифест | `ПрочитатьМанифест` | чтение манифеста из очереди |
| вход, чанк | `ЗначениеИзФайла(...)` | `consume` + `ack` |

Маппинг: чанк = тело сообщения (тот же deflate); durable queue + publisher confirms; `ack` после записи набора = встроенная идемпотентность/докачка; `prefetch` = «Число потоков». Для реального потока — AMQP с **бинарным** телом (чанк уже бинарный, Base64 не нужен: он раздувает тело на ~33%, а HTTP management-API RabbitMQ — плоскость управления, не данных). AMQP-коннектор в поставку не входит — инструмент даёт стабильный формат чанков/манифеста и готовый шов.

## Структура репозитория

```
├── src/        # проект 1C:EDT: .mdo, BSL-модули, форма (.project, .settings; DT-INF не версионируется)
├── build/      # артефакты: БыстрыйПереносРегистров.cfe (расширение), ПереносРегистров.cf (конфигурация)
└── docs/       # УстановкаИЭксплуатация.md (установка, грабли, профили СУБД), АрхитектураИУроки.md (решения и уроки ревью)
```

Ключевые объекты (7 объектов метаданных, самодостаточно):

| Объект | Роль |
|---|---|
| `Обработка.БыстрыйПереносРегистров` | Движок в модуле менеджера (`ВыгрузитьРегистры` / `ЗагрузитьРегистры`) + управляемая форма |
