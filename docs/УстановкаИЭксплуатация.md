# Установка и эксплуатация

Инструмент поставляется как расширение конфигурации `БыстрыйПереносРегистров.cfe`
(префикс `БПР_`, назначение `AddOn`, режим совместимости 8.5.1, собрано на платформе 8.5.1.1343).
Расширение самодостаточно — ноль зависимостей от базовой конфигурации, подключается
к любой целевой конфигурации без правки основной.

---

## 1. Установка расширения `.cfe`

### 1.1. Штатный путь (Конфигуратор)

1. Конфигуратор → **Конфигурация → Расширения конфигурации → Добавить** → выбрать `БыстрыйПереносРегистров.cfe`.
2. **Обязательно** подключить расширение как доверенное: в строке расширения снять галки
   **«Безопасный режим»** и **«Защита от опасных действий»**, затем перезапустить сеанс.
3. Обновить конфигурацию базы данных (`/UpdateDBCfg` при работе из CLI).

### 1.2. Грабля №1: безопасный режим (ключевая)

Если оставить расширение в безопасном режиме, файловый I/O (`ЗначениеВФайл`) падает с ошибкой:

> «Установлен безопасный режим. Выполнение операции запрещено»

Симптом коварный: **запись в регистры работает, а выгрузка в файл — нет**.
Кодом изнутри расширения это не обходится — доверенное подключение является
требованием поставки.

Как снять безопасный режим **без Конфигуратора** (через веб-клиент):

1. Включить «Режим технического специалиста».
2. «Функции для технического специалиста» → «Управление расширениями конфигурации».
3. Выключить **оба** тумблера («Безопасный режим», «Защита от опасных действий»).
4. Нажать «Перезапустить».

### 1.3. Грабля №2: режим совместимости интерфейса 8.5

Интерфейс целевой базы должен быть **8.5** (`interfaceCompatibilityMode = Version8_5`),
иначе форма обработки откроется **пустой** — заголовок есть, тело без элементов.
В БСП-релизах обычно уже так.

Лечение (в исходниках целевой конфигурации, `Configuration.mdo`): перед тегом
`<compatibilityMode>` добавить:

```xml
<interfaceCompatibilityMode>Version8_5</interfaceCompatibilityMode>
<version85InterfaceMigrationMode>Use</version85InterfaceMigrationMode>
```

### 1.4. Грабля №3: файловая ИБ и применение расширения из EDT

Применение расширения к **файловой** ИБ через EDT (`update_database`) упирается в глюк
среды `Файл не обнаружен 'zip:///'` — это дефект EDT/платформы, не расширения
(постороннее расширение, открытое в воркспейсе EDT, тоже его провоцирует). Обходы:

- собрать `.cfe` и подключить через **Конфигуратор** (п. 1.1);
- чистый воркспейс EDT (убрать посторонние расширения);
- клиент-серверная ИБ;
- **автономный сервер**: он обслуживает файловую ИБ по пути и применяет расширения
  штатно. После создания конфигурации запуска `ServerApplication.*` обновление базы
  из EDT работает нормально.

Также учтите: открытая веб-сессия держит блокировку файловой ИБ — перед деплоем
закройте вкладку или используйте динамическое обновление.

### 1.5. Проверка установки

Обработка **«Быстрый перенос регистров»** (подсистема «Сервис») → вкладка
«Бенчмарк и настройки» → **ПолныйТест**. Результат: HTML-отчёт `full_test_report.html`
плюс `*.bin` / `manifest.json` в папке обмена. Полный тест делает round-trip 5 млн
записей на синтетическом регистре со сверкой данных; реальные регистры при этом
**не изменяются** (на них выполняется только выгрузка-чтение).

### 1.6. Интеграция с БСП

Настройка не требуется: фоновые задания запускаются адаптивно — при наличии БСП через
`ОбщегоНазначения.ВыполнитьМетодКонфигурации`, без БСП — через собственный тонкий шим
`ПереносРегистровФон`. Есть также ветвление по типу ИБ (файловая / клиент-серверная).

## 2. Состав объектов расширения

| Объект | Роль |
|---|---|
| `Обработка.БыстрыйПереносРегистров` | Движок в модуле менеджера (`ВыгрузитьРегистры` / `ЗагрузитьРегистры`) + управляемая форма |
| `ОбщийМодуль.ПереносРегистровФон` | Шим — точки входа фоновых заданий (server, внешнее соединение) |
| `ОбщийМодуль.НагрузочныйТест` | Самотест: синтетическое наполнение + бенчмарки + `ПолныйТест` (HTML-отчёт) |
| `РегистрСведений.СинтетическиеДанныеПереноса` | Синтетический широкий РС для самотеста |
| `РегистрСведений.СложныеТипыПереноса` | Фикстура «сложных типов» (UUID, `ХранилищеЗначения`, `ЛюбаяСсылка`, составной тип) для теста round-trip bin/json |
| `Подсистема.Сервис` | Выводит обработку и синтетические РС в командный интерфейс |

Движок работает с **любыми** регистрами (РС/РН/РБ) по именам-параметрам;
тест-харнесс трогает только свой синтетический регистр.

## 3. Профили производительности СУБД

Профиль выбирается на форме и задаёт дефолты размера чанка и числа потоков.
Все дефолты выбраны **по замерам**, не предположениям.

| Профиль | Когда использовать | Что задаёт |
|---|---|---|
| **Файловая** | Файловая ИБ (в т.ч. под автономным сервером) | Чанк в зоне плато ~100–200 тыс. строк (разница внутри плато 25k–200k < 1,5%, деградация на 500k). Выгрузка параллельная; масштабируется до ~4 потоков (упор в read-lock файловой СУБД, не в диск) |
| **MS SQL** | Клиент-серверная ИБ на MS SQL | Чанк **загрузки < 5000 строк** — при превышении порога ~5000 блокировок MS SQL эскалирует блокировки до table lock на всю таблицу регистра |
| **PostgreSQL** | Клиент-серверная ИБ на PostgreSQL | Профиль без MS SQL-ограничения на размер чанка (эскалация блокировок по порогу — специфика MS SQL) |
| **Экстремальный** | Монопольное окно, максимальная скорость | Максимум потоков на выгрузку (масштабируется до 8 потоков, ×2,2 на файловой ИБ); загрузка упирается в запись (~×1,25, плато после 4 потоков) |
| **Вручную** | Нестандартные среды | Все параметры (размер чанка, потоки, сжатие, формат) задаются пользователем. Правило: «мерим, а не предполагаем» — подтверждайте настройки замером |

Дополнительно:

- Загрузка по умолчанию идёт в **1 поток**; хорошо параллелится именно выгрузка (чтение).
- Сжатие (deflate=9) уменьшает транспорт примерно до 15% исходного объёма ценой
  замедления записи ~×3,8 — выгодно при передаче по сети, невыгодно для локального
  клон-синка.

## 4. Режимы загрузки

| Режим | Механика | Когда применять |
|---|---|---|
| **Полная перезаливка** | `ОчищатьПередЗагрузкой = Истина`: регистр очищается, затем грузятся все чанки по манифесту | Полная выгрузка регистра, приёмник должен стать копией источника |
| **С пропуском совпадающих (докачка)** | `ОчищатьПередЗагрузкой = Ложь` + sidecar-файл `_загружено_*.txt`: уже загруженные чанки пропускаются | Повтор после обрыва — см. раздел 5 |
| **Дозапись** | Наборы добавляются к существующим данным без очистки | Частичные выгрузки (отбор по периоду); независимые РС грузятся дозаписью безопасно |

Правила движка:

- **Независимые РС** грузятся дозаписью (безопасно); **подчинённые регистры**
  (РН/РБ, подчинённые документам) — замещением наборов **по регистратору**.
- Отбор по периоду (`НачалоПериода`/`КонецПериода` для РН/РБ и периодических РС)
  фиксируется в манифесте (поле «Отбор»). **Защита:** загрузка частичной выгрузки
  с полной очисткой запрещена движком — используйте «Дозапись»; для РН/РБ замещение
  по регистратору идемпотентно, поэтому повтор безопасен.
- **Запрет микса при отборе по периоду.** Смешивать в одной выгрузке с отбором
  периодические регистры (отбор применяется) и регистры **без поля Период** (отбор
  игнорируется, регистр выгрузился бы целиком) — запрещено: движок падает fail-fast
  **до записи каких-либо файлов** со списком проблемных регистров. Причина: непериодический
  регистр, выгруженный целиком под манифестом частичной выгрузки, при «Дозаписи» удвоился бы
  (дубли). Выгружайте такие регистры отдельной выгрузкой без отбора.

## 5. Докачка после обрыва (`_загружено_*.txt`)

При загрузке рядом с чанками ведётся sidecar-файл `_загружено_<Регистр>.txt` —
журнал уже принятых чанков. Если перенос оборвался:

1. Запустите загрузку повторно с `ОчищатьПередЗагрузкой = Ложь`.
2. Уже загруженные чанки будут пропущены по журналу; докачаются только оставшиеся.

Почему это надёжно:

- запись каждого чанка атомарна;
- для подчинённых регистров замещение по регистратору идемпотентно — повторная
  загрузка того же чанка не дублирует движения.

## 6. Живой прогресс и отмена

Перенос запускается **фоновым заданием**; наблюдение — через файлы в папке обмена:

| Файл | Содержимое |
|---|---|
| `_план.json` | План переноса (что и сколько предстоит) |
| `_прогресс_*.json` | Пишется каждым воркером **после каждого чанка** |
| `_итог.json` | Итог завершённого переноса |

Форма опрашивает файлы прогресса **раз в 2 секунды** и рисует HTML-прогресс
(с анимированным SVG-кроликом). Кнопка **«Отмена»** останавливает фоновый перенос.

Файловый прогресс — это ещё и observability мимо UI: состояние переноса можно
читать с диска любым внешним скриптом, а частичный результат переживает обрыв
длинного веб-вызова.

**Троттлинг записи прогресса.** Прогресс-файл воркер перезаписывает не чаще раза
в 2 секунды (по времени изменения файла) — на мелком чанке (профиль MS SQL, 4000
строк) без троттлинга получались бы тысячи перезаписей мелких json в минуту на поток.
Финальное значение по каждому потоку пишется всегда, поэтому итоговый процент точен.

**Подпапки сеанса `Run_*` (изоляция параллельных прогонов).** Фоновый перенос,
запущенный **с формы**, работает в собственной подпапке `Run_<дата>_<время>_<уид>`
внутри папки обмена. Это закрывает два сценария разом:

- **Отмена не оставляет мусора в следующем прогоне.** Отмена главного фонового задания
  не прерывает каскадно уже запущенные воркеры-диапазоны (платформа их не убивает) —
  они дописывают чанки в свою, уже «мёртвую» подпапку. Следующая выгрузка идёт в новой
  подпапке, и «хвост» отменённого прогона физически не может попасть в её манифест.
- **Две выгрузки/загрузки в одну папку обмена не топчут друг друга** — у каждой свои
  `_прогресс_*` / `_план` / `_итог` / чанки.

Загрузка и «Отправить в RabbitMQ» с формы находят источник сами: `manifest.json` прямо
в папке обмена или **свежайшую** подпапку `Run_*` с манифестом. Прямой вызов API
(`ВыгрузитьРегистры` / `ЗагрузитьРегистры`) папку не подменяет — за изоляцию отвечает
вызывающий (харнесс чистит папку сам).

## 7. Метрики `.prom` и Grafana

Рядом с отчётами инструмент пишет файл **`bpr_transfer.prom`** в формате
OpenMetrics/Prometheus с метриками:

- `bpr_rows_total` — перенесено строк;
- `bpr_seconds` — длительность;
- `bpr_rows_per_second` — скорость;
- `bpr_bytes_total` — объём транспорта (**строго файлы текущего манифеста**: `manifest.json`
  плюс перечисленные в нём чанки, а не «размер папки» — чужие старые чанки в общей папке
  метрику не завышают);
- `bpr_threads` — число потоков.

Перф-матрица форматов (bin / bin+deflate9 / json) дополнительно пишет
`benchmark_format.txt` + `benchmark_format.prom`.

### Подключение к Grafana (node_exporter textfile collector)

1. Запустите `node_exporter` с флагом `--collector.textfile.directory=<каталог>`.
2. Направьте `.prom`-файлы инструмента в этот каталог (пишите папку обмена туда
   или копируйте файл после переноса).
3. Prometheus скрейпит node_exporter штатно; в Grafana метрики `bpr_*` доступны
   как обычные series — стройте панели скорости/объёма по прогонам.

Это сознательно дешёвый путь телеметрии: полноценный HTTP scrape-endpoint отложен
(требует постоянного сервиса), textfile collector закрывает задачу без него.

## 8. Эксплуатационные ограничения (помнить всегда)

- Формат `.bin` (`ЗначениеВФайл`) — **внутренний формат платформы**, версионно-хрупкий:
  может быть несовместим между релизами платформы и при изменении метаданных.
  Страховка — портативный формат `ФорматДанных = JSON` (`СериализаторXDTO`, выгрузка
  в 1 поток, поле «Формат» в манифесте; сжатие в JSON игнорируется).
- JSON решает cross-**версию** платформы, но не cross-**базу**: ссылки остаются GUID
  и между разными базами не маппятся (маппинг ссылок — отдельная задача).
- **JSON и «жирные» типы.** `СериализаторXDTO` в режиме JSON корректно round-трипит
  `УникальныйИдентификатор`, `ХранилищеЗначения` (в т.ч. блоб ~1 МБ), ссылки и составные
  типы — проверено тестом «Сложные типы» (см. кнопку «Тесты ревью v2»). Два нюанса:
  (1) `ХранилищеЗначения` XDTO кодирует в Base64 — на регистрах с крупными хранилищами
  JSON раздувает транспорт и память сериализатора; для таких регистров используйте `.bin`.
  (2) **пустое** поле составного типа / `ЛюбаяСсылка` читается запросом выгрузки как `NULL`
  и так и грузится обратно (было `Неопределено` — стало `Null`): это нормализация «пусто→пусто»,
  не потеря данных (ни XDTO, ни бинарь на этом не падают).
- Отбор по периоду выгружается последовательно, без параллельного сплита (сознательно).
- Валидация пути папки обмена (`ПроверитьБезопасностьПапки`): запрещены пустой и
  относительный путь, корень диска, системные каталоги (`\Windows\`, `\System32\`,
  `\Program Files\`). Расширение исполняется доверенным кодом от имени серверного
  процесса (rphost) — относитесь к папке обмена соответственно.

### 8.1. Безопасность RabbitMQ-транспорта (PoC)

RMQ-транспорт (вкладка «RabbitMQ») — доверенный dev-инструмент; при выносе за пределы
доверенного контура помните:

- **Path traversal из очереди закрыт.** Имя файла из конверта сообщения (`Конверт["Файл"]`)
  валидируется **до записи на диск** (`ПроверитьИмяФайлаКонверта`): непустая строка без
  `\`, `/`, `..`, `:`. Отравленное сообщение (`..\..\Windows\System32\evil.dll`) не пишется
  и **не подтверждается ack** — остаётся в очереди для разбора администратором. Тем не менее
  очередь — недоверенный источник: не подключайте инструмент к очереди, доступной посторонним.
- **Размер чанка для RMQ ≤ 50 000 строк.** Компонента PinkRabbitMQ гоняет строки: чанк
  читается в `ДвоичныеДанные` → Base64 (+33%) → строка UTF-16 (×2) → JSON-конверт. На чанке
  100k строк это десятки-сотни МБ пиковой памяти `rphost`; на 500k возможен OutOfMemory.
  Для потоковой передачи гигабайтов см. размеченный в коде шов бинарного тела AMQP.
- **Путь к внешней компоненте — вектор доверия.** `ПутьККомпоненте` грузит произвольную
  DLL в `rphost` (безопасный режим снят). Убедитесь, что это именно PinkRabbitMQ из доверенного
  источника; не берите dll из папки, доступной на запись посторонним.
- **Пароль RMQ** хранится в реквизите формы **открытым текстом** — норма для PoC/dev.
  Для продакшена храните пароль в безопасном хранилище БСП и передавайте ссылку (opt-in,
  не ломает принцип «ноль зависимостей» базовой поставки).

### Ориентиры производительности (файловая ИБ на NVMe, синтетический РС)

| Объём | Выгрузка | Загрузка | Транспорт | Сверка |
|---|---|---|---|---|
| 1 млн строк | 9,4 с (106k стр/с) | 7,8 с (128k стр/с) | 151,5 МБ | 1:1 OK |
| 5 млн строк | 56,8 с (88k стр/с) | 53,1 с (94k стр/с) | 772,9 МБ (162 Б/стр) | 1:1 OK |

Сводный замер после всех фиксов (5 млн строк): выгрузка 4 потока — 36,3 с (×1,63),
загрузка 4 потока — 38,1 с (131k стр/с), round-trip сверка OK.
