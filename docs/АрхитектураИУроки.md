# Архитектура и уроки

Документ фиксирует ключевые архитектурные решения инструмента «Быстрый перенос регистров» (расширение `БПР_`, назначение AddOn, режим совместимости 8.5.1) и уроки, извлечённые из внешнего ревью и перф-замеров. Принцип, проходящий через весь документ: **мерим, а не предполагаем** — каждое перф-решение подтверждается бенчмарком, дефолты (размер чанка, сжатие, число потоков) выбраны по замерам.

---

## 1. Архитектура

### 1.1. Движок в модуле менеджера обработки

Весь движок переноса живёт в модуле менеджера `Обработка.БыстрыйПереносРегистров` — экспортные точки входа `ВыгрузитьРегистры` / `ЗагрузитьРегистры`. Управляемая форма — только тонкий UI поверх движка.

Почему модуль менеджера, а не модуль объекта или форма:

- движок **stateless**: работает с любыми регистрами (РС/РН/РБ) по именам-параметрам, не хранит состояние экземпляра обработки;
- вызов не требует создания объекта обработки — дёшево дергать из формы, из фоновых заданий, из внешнего кода;
- ноль зависимостей от базовой конфигурации: 6 объектов метаданных, всё самодостаточно, drop-in в любую целевую конфигурацию без правки основной.

Механика переноса: наборы записей сериализуются в бинарные чанки (`ЗначениеВФайл`) + `manifest.json` с описанием порций → транспорт (файлы / шина) → загрузка замещением по регистратору. Параллельная выгрузка/загрузка, опциональное сжатие (deflate).

### 1.2. Шим общего модуля для фоновых заданий

`ОбщийМодуль.ПереносРегистровФон` — тонкий шим с точками входа для фоновых заданий (флаги «Сервер» и «Вызов внешнего соединения»).

**Почему нельзя таргетить модуль менеджера обработки напрямую.** Платформа принимает в качестве метода фонового задания только экспортный метод *общего модуля* — метод модуля менеджера обработки не является допустимой целью `ФоновыеЗадания.Выполнить`. Поэтому нужен посредник: фоновое задание стартует метод шима, а шим уже вызывает движок в модуле менеджера. Шим — одна строка делегирования на точку входа, вся логика остаётся в движке.

Запуск фоновых заданий адаптивен к среде:

- есть БСП → через `ОбщегоНазначения.ВыполнитьМетодКонфигурации`;
- нет БСП → через собственный шим `ПереносРегистровФон`.

Настройка не требуется; дополнительно движок ветвится по типу ИБ (файловая / клиент-серверная).

### 1.3. Чанкование с целостностью регистратора

Для независимых РС чанк — просто N строк (загрузка дозаписью, безопасный путь). Для подчинённых регистров (РН/РБ и РС с регистратором) целостность набора по регистратору обязательна: замещение (`Набор.Записать(Истина)`) по регистратору, разорванному границей чанка, теряет движения.

Решение — с обеих сторон трубы:

- **Выгрузка:** `УПОРЯДОЧИТЬ ПО Регистратор`, рез чанка **только на смене регистратора** — движения одного регистратора никогда не расщепляются между чанками.
- **Загрузка:** **линейный конечный автомат O(n)** по отсортированному потоку строк: идём по строкам, копим набор текущего регистратора, на смене регистратора пишем набор замещением и начинаем следующий. Это вместо наивного `НайтиСтроки` в цикле по регистраторам — O(n²) на больших чанках.

Побочный бонус: замещение по регистратору идемпотентно, что вместе с sidecar-файлом `_загружено_<Регистр>.txt` даёт дешёвую докачку после обрыва (повтор с `ОчищатьПередЗагрузкой = Ложь` пропускает уже загруженные чанки).

Регресс-тест на границу чанка: 25 регистраторов × 7 движений, размер чанка 10 (граница заведомо рвётся) → 183/183 движений, сверка по всем регистраторам OK.

### 1.4. Параллелизм: MIN/MAX-сплит + лог перекоса; откат квантилей

Параллельная выгрузка режет диапазон данных на отрезки по потокам. Реализованы и замерены два варианта:

| Вариант | Идея | Результат (5 млн строк, 4 потока) |
|---|---|---|
| **MIN/MAX-сплит** (принят) | `MIN`/`MAX` границы диапазона + дешёвый `КОЛИЧЕСТВО` на поток; отрезки равной ширины | **×1.63–1.70** |
| Квантили (откачен) | «Идеальные» границы через полный `УПОРЯДОЧИТЬ ПО` по 5 млн строк | **×1.06** |

Квантили теоретически дают ровные по числу строк отрезки, но полная сортировка 5 млн строк на файловой СУБД съела весь выигрыш от параллелизма: ускорение упало с ×1.70 до ×1.06. Откатили на грубый MIN/MAX-сплит — он «неправильный» в теории и быстрый на практике.

Вместо умной балансировки — **наблюдаемость**: в лог пишется реальное число строк на поток, перекос виден сразу. Это осознанный дизайн: балансировку (см. §3) внедрять по факту зафиксированного перекоса, а не спекулятивно.

Пределы масштабирования (5 млн строк, файловая ИБ на NVMe):

- выгрузка: ×1.15 (2 потока) → ×1.90 (4) → ×2.20 (8); практическое плато ~4 потока — упор в read-lock файловой СУБД, не в диск и не в перекос данных;
- загрузка: ~×1.25, плато после 4 потоков — упор в запись; по умолчанию загрузка идёт в 1 поток.

### 1.5. Формат чанка: bin vs json

Базовый формат — `.bin` через `ЗначениеВФайл`: максимально быстрый, но это **внутренний формат платформы** — версионно-хрупкий (может не пережить смену релиза платформы или изменение метаданных). Это главный стратегический риск формата, отсюда — второй формат.

Параметр `ФорматДанных`:

| | 0 = bin | 1 = JSON |
|---|---|---|
| Сериализация | `ЗначениеВФайл` (ТЗ напрямую) | `СериализаторXDTO` / `ЗаписатьJSON`, `НазначениеТипаXML.Явное` |
| Сильная сторона | Скорость (локальный клон-синк) | Портативность: переживает смену версии платформы, читаем глазами |
| Сжатие | Опциональный deflate | Игнорируется |
| Параллелизм выгрузки | Да | 1 поток |

Ветвление по формату — **только в упаковке/распаковке значения**; движок чанкования, замещения по регистратору и параллелизма не трогается. Манифест получает поле `"Формат"` (уже хранит `Версия` и `Сжатие`), загрузка распознаёт формат по расширению файла.

Замер форматов упаковки (1 млн строк):

| Вариант | Запись | Размер | Чтение |
|---|---|---|---|
| A — ТЗ напрямую (`ЗначениеВФайл`) | 1.21 с | 151.5 МБ | 2.09 с |
| B — ХранилищеЗначения без сжатия | 1.71 с | 211.4 МБ (139.6% от A) | 4.06 с |
| C — ХранилищеЗначения deflate=9 | 4.60 с | 22.4 МБ (14.8% от A) | 2.27 с |

Итог: A — быстрее всего локально; C — в ~7 раз компактнее для передачи по сети (цена: запись ×3.82, чтение ×1.09).

Важное ограничение: JSON решает cross-**версию**, но не cross-**базу** — ссылки остаются GUID и между разными базами не маппятся (маппинг ссылок — отдельная задача).

### 1.6. Точки расширения транспорта → RabbitMQ

Транспорт чанков абстрагирован: выгрузка **производит** чанки, загрузка **потребляет**; из коробки транспорт — файлы (`.bin` + `manifest.json` в папке обмена). Все границы транспортного шва помечены в `ManagerModule.bsl` тегом `ТОЧКА РАСШИРЕНИЯ` (находятся грепом); движок при замене транспорта не трогается.

Маппинг на RabbitMQ:

| Граница | Функция движка | RabbitMQ-аналог |
|---|---|---|
| выход, чанк | `ЗаписатьПорцию` / `ЗаписатьПорциюПотока` | `publish` (**routing key = имя регистра**) |
| выход, манифест | `ЗаписатьМанифест` | стартовое сообщение / свойства |
| вход, манифест | `ПрочитатьМанифест` | чтение манифеста из очереди |
| вход, чанк | `ЗначениеИзФайла(...)` | `consume` + `ack` |

Правила маппинга:

- **чанк = тело сообщения** как есть (тот же deflate; чанк уже бинарный — Base64 не нужен);
- **durable queue + publisher confirms** — доставка без потерь;
- **`ack` после записи набора** = встроенная идемпотентность/resume (то же свойство, что у файловой докачки);
- **`prefetch` = «Число потоков»** — то же самое лимитирование конкуренции, что в файловом режиме.

Оговорка к схеме «Base64 + HTTP»: Base64 раздувает тело на ~33%, а HTTP management-API RabbitMQ — плоскость управления, не данных. Для реального потока — AMQP с бинарным телом; HTTP-PoC годится только для демонстрации схемы. AMQP-коннектор (внешняя компонента или готовый RMQ-клиент) — вне поставки: инструмент даёт шов и стабильный формат чанков/манифеста.

## 2. Ключевые уроки внешнего ревью

### 2.1. Критический баг потери данных в подчинённых регистрах

Внешнее ревью нашло **критический баг потери данных** в подчинённых регистрах (РН/РБ) при замещении по регистратору на границе чанка — при том, что все тесты были зелёные.

Причина слепого пятна: тест-харнесс гонял только независимый РС, то есть **только безопасную ветку дозаписи**; ветка замещения по регистратору не покрывалась вовсе — именно в ней и жил баг. «Зелёные тесты» доказывали корректность лишь того пути, который тестировался.

Вывод: покрывать **все** ветки записи, особенно деструктивные (замещение). Ответ — регресс-тест из §1.3, который заведомо рвёт границу чанка внутри регистратора (25 регистраторов × 7 движений, чанк 10 → 183/183 движений OK).

### 2.2. Мерить, а не предполагать

«Теоретически верная» оптимизация (квантильная балансировка через полную сортировку) в реальном замере уронила параллелизм с ×1.70 до ×1.06 — см. §1.4. Правило: перф-правки вносятся **только с замером до/после**; оптимизации внедряются по факту зафиксированного перекоса/узкого места (лог его показывает), не спекулятивно.

### 2.3. Fail-fast до очистки приёмника

Деструктивные операции валидируются **до** того, как что-то стёрто:

- загрузка частичной выгрузки (манифест с отбором по периоду) в режиме полной очистки **запрещена движком** — иначе очистили бы весь регистр, а залили только срез; движок падает с понятной ошибкой до очистки и предлагает режим «Дозапись» (для РН/РБ замещение по регистратору идемпотентно и безопасно само по себе);
- валидация пути (`ПроверитьБезопасностьПапки`): запрет пустого/относительного пути, корня диска, системных каталогов (`\Windows\`, `\System32\`, `\Program Files\`) — расширение исполняется доверенным кодом от имени серверного процесса (rphost).

Общий принцип: сначала все проверки совместимости манифеста, режима и путей — потом первая деструктивная запись.

### 2.4. Второй раунд ревью (v2): болезни распределённых систем

Асинхронный переход v2 (фоновые воркеры, живой прогресс, RabbitMQ-транспорт) принёс классику распределённых систем — гонки, утечки жизненного цикла, векторы через шину. Три MUST-fix, все проверены живым тестом («Тесты ревью v2» на форме):

- **Зомби-воркеры при отмене → изоляция подпапками сеанса.** Отмена главного фонового задания **не убивает каскадно** его воркеры-диапазоны (платформа их не прерывает) — они дописывают чанки в папку. Раньше следующая выгрузка могла подхватить этот «хвост» в свой манифест → битые/задвоенные данные при загрузке. Фикс: каждый фоновый перенос с формы работает в подпапке `Run_<дата>_<время>_<уид>`; хвост отменённого прогона остаётся в своей мёртвой подпапке и в манифест следующего попасть не может. Тот же ход закрывает и общую папку на двоих (у каждого прогона свои служебные файлы). Тест: выгрузка 1 млн → отмена на лету → сразу вторая выгрузка → манифест второй чист, 100 файлов-хвостов изолированы.
- **Path traversal из очереди.** Имя файла из конверта RabbitMQ писалось на диск как пришло — недоверенная очередь могла подсунуть `..\..\Windows\System32\evil.dll` (rphost со снятым безопасным режимом послушно записал бы). Фикс: `ПроверитьИмяФайлаКонверта` валидирует имя **до** записи (запрет `\`, `/`, `..`, `:`); отравленное сообщение не подтверждается `ack` и остаётся в очереди. Урок тот же, что §2.3, но для нового источника недоверенного ввода — шины.
- **Микс регистров при отборе по периоду.** Отбор применяется к периодическим регистрам, но игнорируется для непериодических (те выгрузились бы целиком) — под одним манифестом «частичной» выгрузки. При «Дозаписи» непериодический регистр, выгруженный целиком, задвоился бы. Фикс: движок запрещает такой микс fail-fast до записи файлов (тот же принцип «проверка до деструктива», расширенный на выгрузку).

Наблюдаемость тоже поправлена по замечаниям: троттлинг записи прогресса (не чаще раза в 2 с — против дискового спама на мелком чанке) и честный `bpr_bytes_total` (строго файлы текущего манифеста, а не «размер папки» с чужими старыми чанками).

Отдельно проверен «типовой ад JSON»: синтетический РС с `УникальныйИдентификатор`, `ХранилищеЗначения` (блоб ~1 МБ), `ЛюбаяСсылка` и составным типом round-трипит и через `.bin`, и через JSON (`СериализаторXDTO` не падает). Единственный нюанс — пустое поле составного типа нормализуется `Неопределено→Null` (запрос выгрузки читает пустой составной как `NULL`); это «пусто→пусто», не потеря данных.

## 3. Гистограммная балансировка потоков — идея v3

Текущий MIN/MAX-сплит режет диапазон по **ширине** (равные отрезки дат/ключей) и уже даёт ~×1.6; лог «строк/поток» показывает фактический перекос. На перекошенных данных (типичный случай: последние годы в разы толще ранних) равная ширина даёт неравный объём работы — один поток тащит половину строк.

Идея v3 — резать по равному **весу** строк, а не по ширине диапазона, на основе дешёвой гистограммы. Индексный запрос без сортировки:

```bsl
ВЫБРАТЬ
	ГОД(Т.Период) КАК Год,
	КОЛИЧЕСТВО(*) КАК ЧислоСтрок
ИЗ
	РегистрНакопления.ИмяРегистра КАК Т
СГРУППИРОВАТЬ ПО
	ГОД(Т.Период)
```

Дальше: веса лет → жадно набирать корзины равного суммарного веса → границы корзин становятся границами потоков. В отличие от отвергнутых квантилей здесь нет `УПОРЯДОЧИТЬ ПО` по всей таблице — только `СГРУППИРОВАТЬ ПО` по индексированному полю, то есть цена подготовки не съедает выигрыш.

Сознательно **не реализовано** в v1/v2 — по урокам §2.2:

- на файловой СУБД потолок — read-lock (~4 потока), балансировка там почти ничего не даёт; выигрыш ожидается в основном на серверной СУБД с перекошенными данными;
- включать по факту: лог перекоса уже встроен — когда он покажет реальный перекос на реальных данных, будет и замер «до», против которого проверять «после».
