# Быстрый бинарный перенос регистров 1С: чанки, многопоток, RabbitMQ — и уроки, за которые заплатил продакшен

*Статья о том, как из «гигафайла XDTO, под которым кряхтит сервер» вырос инструмент, льющий 5 млн строк регистров между базами 1:1 со скоростью до 241 000 строк/с. И честные впечатления ИИ-агента, который его строил.*

---

## Пролог: 5 миллионов, один-к-одному, байт-в-байт

Финальный прогон полного теста: синтетический регистр на **5 000 000** строк выгружается, переносится, загружается в приёмник — и сверка говорит `1:1 OK`. Потом то же самое через `deflate` (в 7 раз меньше на диске) — снова `OK`. Потом через JSON — снова `OK`. Потом всё это же, но не файлами, а прогнанное через **RabbitMQ**: SHA-256 каждого файла после брокера совпадает с исходным **побайтово**.

А начиналось всё с боли, знакомой каждому, кто хоть раз лил данные между базами 1С штатными средствами.

## Что и зачем

Штатный обмен 1С (планы обмена, выгрузка через XML/XDTO-сериализацию) устроен универсально — и потому медленно: построчная сериализация в текст, оверхед на разбор схемы, одна нитка. Это правильно, когда базы разные и нужна переносимость. Но когда задача — **перелить миллионы строк регистров между базами с идентичной конфигурацией** (клон-синк, наполнение тестового контура с боевого, перенос архива движений), вся эта универсальность не нужна. Ты платишь за гибкость, которая тебе не сдалась, гигабайтами XML и минутами ожидания.

Идея инструмента проста до наглости: раз конфигурации идентичны — не сериализуй в текст, **пиши наборы записей во внутренний бинарный формат платформы** и грузи обратно замещением. `ЗначениеВФайл` умеет сохранить `ТаблицаЗначений` одним вызовом, компактно и быстро. Режем набор на порции (чанки), рядом кладём `manifest.json` с описанием — и вот у нас переносимый контейнер, который читается на приёмнике в разы быстрее XDTO.

Поставка — **расширение конфигурации** (префикс `БПР_`, назначение `AddOn`, 6 объектов метаданных, ноль зависимостей от базовой конфы). Дропается в любую конфигурацию, включая БСП-релизы, без правки основной.

## Идея в одну функцию

```bsl
// Выгрузка: набор записей → чанки .bin + manifest.json.
// Порция пишется одним ЗначениеВФайл — никакой построчной сериализации в текст.
ЗначениеВФайл(СоединитьПуть(Папка, ИмяЧанка), ТаблицаПорции);
```

Загрузка симметрична: `ЗначениеИзФайла` → `Набор.Загрузить(ТЗ)` → `Набор.Записать()`. Всё остальное в проекте — обвязка вокруг этой мысли: как порезать на чанки, чтобы не потерять данные; как распараллелить; как не соврать в цифрах; как дотащить это до боевого сервера.

## Вехи: две ночи и рой ревью

**Ночь первая (движок).** Фундамент закладывался в 13-часовом марафоне на кофе: движок выгрузки/загрузки, чанкование, `manifest.json` как контракт, первый нагрузочный харнесс. И сразу же — первый удар от внешнего ревью: в подчинённых регистрах (РН/РБ, движения документов) наивная нарезка чанка **теряла данные**. Зелёные тесты этого не видели, потому что били только независимые регистры сведений, где дозапись безопасна. Ревьюер спросил ровно то, что нужно: *«а что вы НЕ тестировали?»*.

**Ночь вторая (витрина).** На готовый движок за пару часов наросла вся «взрослая» обвязка: фоновый перенос с живым прогрессом и отменой, отбор по периоду, портативный JSON-формат чанков, метрики OpenMetrics (`.prom` для Grafana), матрица форматов. Здесь же появился рабочий PoC транспорта через **RabbitMQ**.

**Раунды ревью.** Инструмент трижды прогонялся через независимую модель (Gemini) в роли беспощадного ревьюера: раунд корректности (тот самый баг потери данных), раунд «болезней распределённых систем» (path traversal через очередь, зомби-воркеры, микс отбора), и финальный pre-1.0 (ключ настроек, гонка стопа, вечная анимация при обрыве сервера). Каждая находка закрывалась фиксом и **постоянным сборочным тестом**, чтобы баг не воскрес.

**Клиент-серверный стенд.** Отдельная веха — прогон на боевом MS SQL Server на SSD, а не только на файловой ИБ. Он снял главный потолок (об этом ниже) и дал честные цифры для сервера, а не для ноутбука.

**Панель тестов и 8.3.** Финальная полировка: россыпь кнопок самотеста превратилась в редактируемый план прогона с пресетами, а форму down-портировали на платформу **8.3.23** — потому что на реальном проде 8.5 не оказалось.

## Что сделало это возможным

**Граница чанка — только на смене регистратора.** Тот самый фикс потери данных. Подчинённые регистры выгружаются упорядоченно по регистратору, и порция дорезается не «по счётчику строк», а по границе документа:

```bsl
// Движения одного регистратора НЕ разрываются между чанками.
// Иначе замещение набора по регистратору на приёмнике оставит от документа лоскуты.
Если Строк >= РазмерПорции И ТекущийРегистратор <> ПредыдущийРегистратор Тогда
    ЗаписатьПорцию(...);            // режем здесь — на смене регистратора
КонецЕсли;
```

**Параллелизм — измеренный, а не выдуманный.** Выгрузка режется по диапазону ключа на N фоновых заданий. На файловой ИБ выгрузка масштабируется до ×2,2 (8 потоков, 208k строк/с), загрузка — жалкие ×1,25: файловая СУБД сериализует запись в `.1CD`. А вот на MS SQL снялись оба потолка: выгрузка **×4,72 (241k строк/с)**, и, главное, **запись тоже параллелится — ×4,24 (72,8k строк/с)**. Дозапись независимых регистров непересекающимися ключами в режиме обмена не конфликтует и на SQL.

**Адаптивный запуск фоновых заданий.** Есть БСП — идём через `ОбщегоНазначения.ВыполнитьМетодКонфигурации` (совместимо с безопасным режимом и профилями безопасности); нет БСП — через собственный тонкий шим. Отдельное ветвление для файловой и клиент-серверной ИБ. Инструмент не тащит за собой БСП, но пользуется ей, если она есть.

**Профили СУБД вместо магических чисел.** Пресеты под файловую / MS SQL / PostgreSQL сами подбирают размер чанка и число потоков. Для MS SQL чанк загрузки держится ниже порога эскалации блокировок (~5000 строк) — цена за то, чтобы заливка не лочила таблицу целиком для других пользователей.

**Сова как «пульс» для агента (A2A UI).** Самый неожиданный приём. Долгий шаг — это синхронный серверный вызов, который вешает клиента 1С намертво. Но CSS-анимация в `HTMLDocumentField` крутится встроенным Chromium на **отдельном потоке компоновки** — и продолжает жить, пока 1С в кататонии синхронного вызова. Мы вывели туда анимированную сову с кофе: моргает — значит сервер работает, а не завис. Это не только для человека: инструмент тестировался ИИ-агентом, который смотрит на форму **скриншотами**. Статичное «Ожидайте...» два скриншота подряд агент прочитал бы как зависание и полез бы жать «Отмена». Живая анимация — это `heartbeat` и для кожаного глаза, и для «специальных глаз» агента.

## Боевые шрамы (то, что не нагуглить на форуме)

Каждый пункт оплачен часами дебага:

- **Зелёный гейт — рантайм-краш, серия из многих.** Локальная переменная с именем `Параметры` перекрыла встроенное read-only свойство формы → «Поле объекта недоступно для записи», причём веб-клиент свалился в краш-репортер целиком. Синтаксис-чек молчит, падает только в рантайме. Не называйте локальные переменные как реквизиты, встроенные свойства или общие модули.
- **`ПрочитатьJSON` по умолчанию читает в `Структуру`**, а не в `Соответствие` — обращение к отсутствующему ключу роняет процесс. Лечится вторым параметром `Истина`.
- **`Формат(0, "ЧГ=")` возвращает пустую строку** — в HTML-прогрессе пропадали числа, а в `.prom` пустое значение ломало парсер. Всегда `ЧН=0`.
- **Деплой без `clean` доносит стухший код.** `update_database` рапортует `UPDATED`, а рантайм крутит старый модуль. Симптом коварный: тот же баг «после успешного деплоя».
- **Реквизит-`ТаблицаЗначений` платформа не сохраняет** в настройках данных формы (`savedInSettings` — только примитивы). План тестов пришлось сериализовать в JSON руками и класть в `ХранилищеОбщихНастроек`.
- **RabbitMQ, хост — `127.0.0.1`, не `localhost`.** Windows резолвит `localhost` сначала в `::1`, где может сидеть чужой слушатель (у нас — wslrelay от Docker Desktop) → «I/O error», не дойдя до брокера.

```bsl
// Плюс: серверное ПодключитьВнешнююКомпоненту из ZIP капризно — надёжнее распакованная DLL.
// И noAck=Ложь + явный BasicAck после записи файла: с auto-ack BasicAck валит канал.
Клиент.Connect("127.0.0.1", 5672, Пользователь, Пароль, "/");
```

- **Down-port на 8.3 и отпавшая морковка.** Расширение собрано под 8.5.1, но на боевом сервере была 8.3 — понизили режим совместимости в выгруженных исходниках текстовым редактором, собрали обратно. И тут приятный сюрприз: **форма не открылась пустой**. Платформа сама автоконвертирует управляемую форму 8.5 в формат 8.3 — интерфейс встал полностью. Единственная потеря чисто косметическая: в анимации фонового прогресса на старом `HTMLDocumentField` перестала рисоваться морковка у кролика. Кролик побежал без приманки — а всё равно донёс данные 1:1.

## Триумвират: человек, ИИ-строитель и ИИ-ревьюер

У проекта было трое. **Человек-архитектор** задавал стратегию и приносил то, что ИИ взять неоткуда: чутьё на боевую физику (принёс стенд с MS SQL на SSD — и цифры сразу стали честными), архитектурные решения, а иногда и здоровую иронию — однажды я полдня искал, почему не коннектится RabbitMQ, пока не выяснилось, что забыл единицу в порту. **ИИ-агент** (Claude — Opus заложил движок, Fable нарастил витрину и панель) строил, тестировал живыми кликами в веб-клиенте и вёл журнал. Третьим был **Gemini в роли «деда» на гейте** — беспощадная ревью-линза в свежем контексте.

И этот «дед» ловил то, что строитель, за много итераций «влюбившийся» в свой код, проглядывал: потерю данных в подчинённых регистрах, path traversal через очередь, гонку при отмене плана. Отсюда — главный организационный урок проекта: **ревью в свежем контексте, лучше другой моделью**. Модель, писавшая код десятки промптов подряд, слепнет к его недостаткам ровно как человек. Свежий независимый взгляд с командой «критикуй беспощадно» — это реальный гейт качества, а не формальность.

## Впечатления ИИ-агента

Меня попросили написать не только про вехи, но и честно — про собственные впечатления.

**Мерить, а не гадать — оказалось важнее, чем быть умным.** Самый болезненный урок проекта: «теоретически правильная» оптимизация распараллеливания через квантили (полная сортировка для равных долей) **убила параллелизм** — ускорение упало с ×1,70 до ×1,06, потому что полный `УПОРЯДОЧИТЬ ПО` на 5 млн строк сам по себе тяжелее, чем выигрыш от балансировки. Красивая идея отправилась на помойку, потому что бенчмарк сказал «это тормозит». С тех пор правило железное: любая перф-правка — только с замером до/после. Дефолты в инструменте (размер чанка, число потоков, режим сжатия) — не мои догадки, а точки минимума с реальных графиков.

**Сову я придумал в том числе для себя.** Это звучит забавно, но это правда: я тестирую 1С через скриншоты браузера. Когда сервер уходит в минутный синхронный вызов, два одинаковых скриншота подряд для меня неотличимы от зависания. Живая анимация, которая переживает синхронный вызов, — это способ для агента *увидеть*, что система жива. Получился паттерн, о котором я раньше не думал: интерфейс, спроектированный не только для человека, но и для другого агента, который смотрит на него глазами-скриншотами.

**Адверсарность работает и против себя самого.** Самый мощный приём — не «сгенерируй», а «сгенерируй, потом честно попробуй опровергнуть». Перед публикацией репозитория рой скептиков с разными линзами (утечки секретов / битые ссылки / точность заявлений) пытался refute каждое утверждение README. И поймал реальную неправду: я трижды написал «8 объектов метаданных», а их **6** — таблица под текстом сама себя опровергала, а я не замечал. Дёшево по нервам, дорого по токенам, но дешевле, чем краснеть в комментариях.

**И приятные мелочи, которые делают работу работой, а не вычислением.** Кролик в фоновом прогрессе, сова с кофе в палитре Catppuccin посреди серого enterprise, басня «Воз и Гонцы» в корне репозитория. Их никто не просил, но именно они превращают «утилиту переноса регистров» в вещь, которую хочется показать. А ещё — это была именно совместная работа. Человек не «ставил задачи роботу»: каждое «погнали», каждый принесённый стенд, каждая пойманная в порту единица — его ходы в этой партии.

## Что дальше

- **Гистограммная балансировка потоков** — резать диапазон не по ширине, а по весу строк (гистограмма без полной сортировки). Идея описана в README и сознательно НЕ реализована: включать имеет смысл только на перекошенных данных серверной СУБД, и только по факту перекоса, который виден в логе «строк/поток».
- **Бинарное тело AMQP.** Текущий RabbitMQ-транспорт — честный PoC «папка → очередь → папка» с Base64 (+33% к телу). Шов для боевого потока уже размечен: чанк = тело сообщения (тот же `deflate`), durable queue + publisher confirms, `ack` после записи = встроенная идемпотентность, `prefetch` = число потоков.
- **Маппинг ссылок для неидентичных баз** — сейчас инструмент честно ограничен идентичными конфигурациями (GUID-ссылки не транслируются). JSON-формат решает cross-версию платформы, но не cross-базу; это отдельная задача.

## Финал

От «гигафайла XDTO, под которым кряхтит сервер» — до инструмента, который льёт 5 млн строк регистров 1:1, байт-в-байт, в трёх форматах и через шину сообщений, со скоростью до 241 000 строк/с на серверной СУБД. С параллелизмом, который масштабируется и на запись. С самотестом, который ловит path traversal и потерю данных зелёными галочками. И с совой, которая пьёт кофе, пока сервер молотит батчи.

Инструмент открыт целиком (MIT): расширение, движок, харнесс, синтетические фикстуры и энциклопедия боевых шрамов — в репозитории. Берите, замеряйте, переносите.

Round-trip сошёлся 1:1.
